# Dictionary / Map — інтерфейс та реалізація в різних мовах

---

# 1️⃣ Dictionary — це інтерфейс

Dictionary (Map) — це абстрактна структура даних типу:

КЛЮЧ → ЗНАЧЕННЯ

Гарантує:

- унікальні ключі
- швидкий пошук
- вставку
- видалення
- перебір

⚠️ Це ЛОГІЧНА модель.
Реалізація залежить від мови.

---

# 2️⃣ Базові операції

- put(key, value)
- get(key)
- remove(key)
- contains(key)
- iteration

Середня складність:
O(1)

---

# 3️⃣ Загальна ідея hash-реалізації

1. key → hash(key)
2. hash → індекс у масиві
3. зберігаємо (key, value)

index = hash % size

# 4️⃣ Реалізації в різних мовах

---

# 🐍 Python

Тип: dict

## Внутрішня структура:

ДИНАМІЧНИЙ МАСИВ + HASH + OPEN ADDRESSING

### Що це означає:

- є один великий динамічний масив
- кожна комірка зберігає:
  (hash, key, value)
- при колізії використовується open addressing
- пошук відбувається шляхом "пробування" наступних слотів

### Схема:

0 | 1 | 2 | 3           | 4 | 5
--|---|---|-------------|---|---
  |   |   | (cat,5)     |   | (dog,3)

📌 Python = Dynamic Array + Hash + Open Addressing

Немає linked list.
Немає дерев.
Все лежить в одному масиві.

---

# ☕ Java (HashMap) — спрощена модель

Інтерфейс: Map<K,V>

## Внутрішня структура:

МАСИВ + LINKED LIST


### Що відбувається:

1️⃣ Є масив (table[])
2️⃣ Кожна комірка масиву — це початок linked list
3️⃣ Якщо виникає колізія — елемент додається у список

### Схема:

table[3]:

(cat,5) → (dog,3) → (fish,8)

📌 Java = Array + Linked List

⚠️ Примітка:
У сучасній Java при великій кількості колізій
linked list може перетворюватися на Red-Black Tree,
але базово — це масив + список.

---

# 🟦 Go (Golang 1.24+) — Swiss Table

Тип: map[key]value

Починаючи з Go 1.24 реалізація map базується на Swiss Table.

## Внутрішня структура:

МАСИВ + OPEN ADDRESSING + GROUPS (Swiss Table)

Swiss Table — це сучасна оптимізована хеш-таблиця.

### Основні особливості:

- відкритий метод адресації (open addressing)
- таблиця розбита на групи (наприклад по 8 слотів)
- кожна група має контрольні байти
- швидкий пошук через перевірку групи одразу
- краще використання кешу процесора

### Схема (спрощено):

Group 0:
[ key | key | empty | key | empty | empty | key | empty ]

Group 1:
[ empty | key | key | empty | empty | empty | empty | empty ]

Тобто:

- немає linked list
- немає класичних bucket overflow
- все працює через open addressing
- але не простий linear probing, а оптимізована групова перевірка

📌 Go = Swiss Table (Array + Open Addressing + Groups)

# 5️⃣ Порівняння реалізацій

| Мова   | Базова структура                     | Колізії вирішуються через |
|--------|---------------------------------------|----------------------------|
| Python | Динамічний масив                     | Open Addressing            |
| Java   | Масив                                | Linked List                |
| Go     | Масив (Swiss Table, групи слотів)    | Open Addressing (group-based) |

Python:

# 7️⃣ Головний педагогічний висновок

Dictionary — це інтерфейс:

КЛЮЧ → ЗНАЧЕННЯ

Але реалізація може бути:

Python → Dynamic Array + Hash  
Java → Array + Linked List  
Go → Swiss Table  

І саме внутрішня реалізація визначає:

- поведінку при колізіях
- використання пам’яті
- швидкість у гіршому випадку
- роботу з кешем процесора
